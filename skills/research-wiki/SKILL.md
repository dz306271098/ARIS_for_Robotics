# Research Wiki: Persistent Research Knowledge Base

Subcommand: **$ARGUMENTS**

## Overview

The research wiki is a persistent, per-project knowledge base that accumulates structured knowledge across the entire ARIS research lifecycle. Unlike one-off literature surveys that are used and forgotten, the wiki **compounds** — every paper read, idea tested, experiment run, and review received makes the wiki smarter.

Inspired by [Karpathy's LLM Wiki pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f): compile knowledge once, keep it current, don't re-derive on every query.

## Core Concepts

### Six Entity Types

| Entity | Directory | Node ID format | What it represents |
|--------|-----------|---------------|--------------------|
| **Paper** | `papers/` | `paper:<slug>` | A published or preprint research paper |
| **Idea** | `ideas/` | `idea:<id>` | A research idea (proposed, tested, or failed) |
| **Experiment** | `experiments/` | `exp:<id>` | A concrete experiment run with results |
| **Claim** | `claims/` | `claim:<id>` | A testable scientific claim with evidence status |
| **Principle** | `principles/` | `principle:<slug>` | A distilled, domain-agnostic principle (via 5-layer protocol in `shared-references/principle-extraction.md`) |
| **Failure-pattern** | `failures/` | `failure-pattern:<slug>` | A distilled, domain-agnostic failure anti-pattern (via 5-layer protocol in `shared-references/failure-extraction.md`) |

### Typed Relationships (`graph/edges.jsonl`)

| Edge type | From → To | Meaning |
|-----------|-----------|---------|
| `extends` | paper → paper | Builds on prior work |
| `contradicts` | paper → paper | Disagrees with results/claims |
| `addresses_gap` | paper\|idea → gap | Targets a known field gap |
| `inspired_by` | idea → paper | Idea sourced from this paper |
| `tested_by` | idea\|claim → exp | Tested in this experiment |
| `supports` | exp → claim\|idea | Experiment confirms claim |
| `invalidates` | exp → claim\|idea | Experiment disproves claim |
| `supersedes` | paper → paper | Newer work replaces older |
| `embodies_principle` | paper\|idea → principle | Paper / idea embodies (or is motivated by) the principle |
| `shares_principle_with` | principle ↔ principle | Two principles are isomorphic or generalize to a common ancestor |
| `failure_mode_of` | failure-pattern → principle | This failure pattern afflicts methods embodying this principle |
| `manifested_as` | idea\|exp → failure-pattern | This idea / experiment exhibited this failure pattern |
| `resolved_by` | principle → failure-pattern | This principle provably resolves (or mitigates) this failure pattern |

Edges are stored in `graph/edges.jsonl` only. The `## Connections` section on each page is **auto-generated** from the graph — never hand-edit it.

## Wiki Directory Structure

```
research-wiki/
  index.md               # categorical index (auto-generated)
  log.md                 # append-only timeline
  gap_map.md             # field gaps with stable IDs (G1, G2, ...)
  query_pack.md          # compressed summary for /idea-creator (auto-generated, max 8000 chars)
  AUDIT_REPORT.md        # contradictions, trends, principle coverage (auto-generated by `audit` subcommand)
  papers/
    <slug>.md            # one page per paper
  ideas/
    <idea_id>.md         # one page per idea
  experiments/
    <exp_id>.md          # one page per experiment
  claims/
    <claim_id>.md        # one page per testable claim
  principles/
    <slug>.md            # one page per distilled principle
  failures/
    <slug>.md            # one page per failure anti-pattern
  graph/
    edges.jsonl          # materialized current relationship graph
```

## Subcommands

### `/research-wiki init`

Initialize the wiki for the current project:

1. Create `research-wiki/` directory structure
2. Create empty `index.md`, `log.md`, `gap_map.md`
3. Create empty `graph/edges.jsonl`
4. Log: "Wiki initialized"

### `/research-wiki ingest "<paper title>" — arxiv: <id>`

Add a paper to the wiki:

1. **Fetch metadata** — use arXiv/DBLP/Semantic Scholar to get full metadata
2. **Generate slug** — `<first_author_last_name><year>_<keyword>` (e.g., `chen2025_factorized_gap`)
3. **Check dedup** — if `paper:<slug>` already exists, update instead of creating
4. **Create page** — `papers/<slug>.md` with full schema (see below)
5. **Extract relationships** — scan the paper's related work / method for connections to existing wiki pages
6. **Add edges** — append to `graph/edges.jsonl`
7. **Update index** — regenerate `index.md`
8. **Update gap_map** — if the paper reveals new gaps or addresses existing ones
9. **Rebuild query_pack** — regenerate `query_pack.md`
10. **Log** — append to `log.md`

  doi: null

### `/research-wiki upsert_principle <slug> — from: paper:<slug>`

Add or update a distilled principle extracted via `shared-references/principle-extraction.md`:

1. **Verify extraction completeness** — the 5-layer output must exist (surface, principle, generalization, adaptation, anti-copy).
2. **Generate principle slug** — `<kebab-case short name>` (e.g., `selective-feature-weighting`, `receding-horizon-replanning`).
3. **Check dedup** — if `principle:<slug>` already exists, add the source paper to `evidence_papers[]` instead of creating a duplicate. Detect near-duplicates via Layer 2 (one-sentence principle) cosine similarity — if a principle with ≥ 0.85 similarity already exists, merge.
4. **Create page** — `principles/<slug>.md` with the schema below.
5. **Add edges** — `embodies_principle` from source paper → principle. If the new principle resembles an existing one (≥ 0.70 similarity), add `shares_principle_with`.
6. **Update index** + regenerate `query_pack.md` (principles section added below).
7. **Log** the upsert.

**Principle page schema:**

```markdown
---
type: principle
node_id: principle:<slug>
name: "<one-sentence principle>"
generalized_form: "<domain-agnostic one-sentence statement>"
evidence_papers: [paper:chen2025, paper:smith2024, ...]
tested_in_projects: []          # slugs of projects (if cross-project) or idea:<id> references
status: EXTRACTED | APPLIED | SUPERSEDED | INVALIDATED
tags: [tag1, tag2]
origin_skill: research-lit | deep-innovation-loop | manual
created_at: 2026-04-07T10:12:00Z
updated_at: 2026-04-07T10:12:00Z
---

# One-sentence principle

[Layer 2 output from principle-extraction.md — WHY the underlying mechanism works, abstracted from all implementation details]

## Generalized Form

[Layer 3 output — domain-agnostic formulation]

## Surface Methods That Embody It

[One line per evidence paper: paper:<slug> — "<surface method description>"]

## Adaptations Tested

[Record every time this principle has been translated into our problem — per Layer 4]
- idea:<id> — round N of project <slug> — outcome: tested-positive | tested-negative | mixed | unexplored

## Anti-Copying Guard

[Per Layer 5 — what NOT to copy when this principle is applied]

## Connections

[AUTO-GENERATED from graph/edges.jsonl — do not edit manually]
```

**Deduplication is load-bearing.** Without dedup, the principle library inflates with near-synonyms; with dedup, it becomes a finite, queryable library of distinct ideas.

### `/research-wiki upsert_failure-pattern <slug> — from: paper:<slug>|idea:<id>|exp:<id>`

Add or update a failure anti-pattern extracted via `shared-references/failure-extraction.md`. Symmetric to `upsert_principle`.

1. **Verify extraction completeness** — the 5-layer output must exist (surface failure, underlying trigger, generalization, adaptation check, resolution status).
2. **Generate failure slug** — `<kebab-case short name>` (e.g., `recurrent-error-accumulation`, `cold-start-data-scarcity`).
3. **Check dedup** — if `failure-pattern:<slug>` already exists, add the source to `evidence_papers[]` / `manifested_in_ideas[]` / `manifested_in_experiments[]` instead of creating a duplicate. Detect near-duplicates via Layer 3 (generalized form) cosine similarity — if a failure with ≥ 0.85 similarity already exists, merge.
4. **Create page** — `failures/<slug>.md` with the schema below.
5. **Add edges**:
   - `failure_mode_of` (failure-pattern → principle) for every principle in `Affects principles`
   - `resolved_by` (principle → failure-pattern) for every principle in `Resolved by principles`
   - `manifested_as` (idea|experiment → failure-pattern) if source is an idea or experiment
6. **Update index** + regenerate `query_pack.md` (top unresolved failures section).
7. **Log** the upsert.

**Failure-pattern page schema:**

```markdown
---
type: failure-pattern
node_id: failure-pattern:<slug>
name: "<short descriptive name>"
generalized_form: "<domain-agnostic one-sentence condition>"
evidence_papers: [paper:<slug>, ...]
manifested_in_ideas: [idea:<id>, ...]
manifested_in_experiments: [exp:<id>, ...]
affects_principles: [principle:<slug>, ...]
resolved_by_principles: [principle:<slug>, ...]
status: active | resolved | theoretical
tags: [scalability, data-efficiency, optimization, generalization, inference, reproducibility, specification, evaluation]
origin_skill: research-lit | deep-innovation-loop | auto-review-loop | experiment-bridge | manual
created_at: ...
updated_at: ...
---

# [Name]

## Underlying Trigger

[Layer 2 from failure-extraction.md — mechanistic cause, domain-agnostic]

## Generalized Conditions

[Layer 3 — conditions under which this fires for ANY method]

## Evidence

[Per source: paper, idea, or experiment — one line with the surface failure observed]

## Resolution Attempts

[Principles or methods reported to resolve it; or "no known resolution"]

## Connections

[AUTO-GENERATED from graph/edges.jsonl — do not edit manually]
```

**Deduplication is load-bearing.** Same argument as principles — without dedup the failure library inflates; with dedup it becomes a queryable cross-project library.

### `/research-wiki query "<topic>"`

Generate `query_pack.md` — a compressed, context-window-friendly summary:

**Fixed budget (max 8000 chars / ~2000 tokens):**

| Section | Budget | Content |
|---------|--------|---------|
| Project direction | 300 chars | From CLAUDE.md or RESEARCH_BRIEF.md |
| Top 5 gaps | 1100 chars | From gap_map.md, ranked by: unresolved + linked ideas + failed experiments |
| Principle library | 1100 chars | Top principles by (evidence_papers count + tested_in_projects count); flag UNEXPLORED principles (no `tested_in_projects` entries) |
| Top unresolved failures | 1000 chars | 5-8 failure-patterns with `status=active` + `resolved_by_principles=[]`, ranked by (evidence_papers × manifestations); treat as sharp ideation seeds |
| Paper clusters | 1000 chars | 3-5 clusters by tag overlap, 2-3 sentences each |
| Failed ideas | 1200 chars | **Always included** — highest anti-repetition value |
| Top papers | 1400 chars | 8-12 pages ranked by: linked gaps, linked ideas, centrality, relevance flag |
| Active chains | 500 chars | limitation → opportunity relationship chains |
| Open unknowns | 400 chars | Unresolved questions across the wiki |

**Pruning priority** (when over budget): low-ranked papers > cluster detail > chain detail. **Never prune** failed ideas, top gaps, UNEXPLORED principles, or Top unresolved failures first (these are the most load-bearing for ideation).

**Key rule:** Read from short fields only (frontmatter, one-line thesis, gap summary, failure note). Do not summarize full page bodies every time.

### `/research-wiki update <node_id> — <field>: <value>`

Update a specific entity:

```
/research-wiki update paper:chen2025 — relevance: core
/research-wiki update idea:001 — outcome: negative
/research-wiki update claim:C1 — status: invalidated
```

After any update: rebuild `query_pack.md`, update `log.md`.

### `/research-wiki lint`

Health check the wiki:

1. **Orphan pages** — entities with zero edges
2. **Stale claims** — claims with `status: reported` older than 14 days
3. **Contradictions** — claims with both `supports` and `invalidates` edges
4. **Missing connections** — papers sharing 2+ tags but no explicit relationship
5. **Dead ideas** — `stage: proposed` ideas that were never tested
6. **Sparse pages** — pages with 3+ empty sections

Output a `LINT_REPORT.md` with suggested fixes.

### `/research-wiki audit`

Deterministic, LLM-assisted pass producing `AUDIT_REPORT.md`. This surfaces the cross-paper intelligence that would otherwise stay latent in the graph. Safe to run any time; idempotent.

**Three analyses:**

**1. Contradictions.**
- Primary source: all `contradicts` edges in `graph/edges.jsonl`.
- Heuristic augmentation: scan paper pages for `## Key Results` sections mentioning the same metric keyword with opposite direction (e.g., "improves" vs "degrades" the same metric). Ambiguous pairs get adjudicated by `codex exec --sandbox read-only -m gpt-5.4` reading both pages directly.
- Output table: `| metric | paper A claim | paper B claim | resolution status | next action |`.
- `resolution status`: OPEN | RECONCILED | OBSOLETE.

**2. Trends.**
- Bucket papers by year (last 5 years).
- For each principle in `principles/`, count `embodies_principle` edges per year.
- Flag rising (≥ 3-year monotonic increase) and declining (≥ 3-year monotonic decrease) principles. Mann-Kendall trend test if ≥ 5 years of data.
- Output: `| principle | year counts | trend | leading papers |`.

**3. Principle coverage.**
- Each principle's `tested_in_projects[]` count and `evidence_papers[]` count.
- Flag **latent opportunities**: principles with `evidence_papers[]` ≥ 3 but `tested_in_projects[]` = 0 (cited by multiple papers, never tried in any of OUR projects).
- Flag **narrow validation**: principles with `tested_in_projects[]` = 1 and outcome = positive (promising but single-project; candidate for cross-project transfer).
- Output: `| principle | evidence count | tested count | status | recommendation |`.

**4. Unresolved failure patterns → ideation seeds.**
- Each failure pattern's `evidence_papers[]`, `manifested_in_ideas[]`, `manifested_in_experiments[]`, `resolved_by_principles[]` counts.
- Flag **unresolved failures**: `status=active` AND `resolved_by_principles=[]` AND `evidence_papers` ≥ 3. These are the sharpest ideation seeds — "repeatedly observed, nobody has solved."
- Flag **failure clusters**: groups of 2+ failure patterns whose `affects_principles[]` overlap by ≥ 50%. Point to meta-failure-patterns spanning many principles (e.g., cold-start data scarcity across learning paradigms).
- Flag **resolution candidates**: failures with exactly 1 `resolved_by_principles` entry AND that principle has `tested_in_projects[]` = 0 (reported resolution, never verified in our projects — candidate for confirmation experiments).
- Output: `| failure pattern | evidence count | unresolved? | overlapping cluster | recommendation |`.

**Auto-invocation:**
- `/idea-creator` Phase 0 (pre-ideation) — read `AUDIT_REPORT.md` to surface latent-opportunity principles AND unresolved failures as ideation seeds.
- `deep-innovation-loop` Phase C (variant design) — read `AUDIT_REPORT.md` failure-cluster section to check whether the proposed variant sits in a known failure cluster.
- `deep-innovation-loop` Phase E Step 2.9 (Trajectory Reanalysis at rounds 15/30/45) — read `AUDIT_REPORT.md` to contextualize which early principle commitments now look narrow or prone to documented failures.
- `auto-review-loop` Phase C.5.1 (Failure Archaeology) — read `AUDIT_REPORT.md` to check for known contradictions around the current fix direction AND match against failure patterns before external literature search.

**Implementation note.** The heuristic + LLM-adjudication pattern keeps the audit cheap (< 5 minutes for a 50-paper wiki) and deterministic (same inputs → same report).

## Integration with Existing Workflows

### Hook 1: After `/research-lit` finds papers

```
# At end of research-lit, after synthesis:
if research-wiki/ exists:
    for paper in top_relevant_papers (limit 8-12):
        /research-wiki ingest paper
        for each gap identified:
            add_edge(paper.node_id, gap_id, "addresses_gap")
        for each explicit relation to existing wiki paper:
            add_edge(paper.node_id, target.node_id, relation_type)

    # NEW: mandatory principle + failure-pattern extraction for top 5-8 papers (research-lit Step 2)
    for paper in top_relevant_papers (limit 5-8, highest relevance):
        # Single Codex call extracts BOTH principles and failures from the paper
        apply shared-references/principle-extraction.md 5-layer protocol  → principles
        apply shared-references/failure-extraction.md  5-layer protocol  → failure patterns (from Limitations + negative ablations)

        for each principle:
            /research-wiki upsert_principle <slug> — from: paper:<slug>
            add_edge(paper.node_id, principle.node_id, "embodies_principle")

        for each failure-pattern:
            /research-wiki upsert_failure-pattern <slug> — from: paper:<slug>
            for affects_principle in failure.affects_principles:
                add_edge(failure.node_id, affects_principle, "failure_mode_of")
            for resolved_by_principle in failure.resolved_by_principles:
                add_edge(resolved_by_principle, failure.node_id, "resolved_by")

    rebuild query_pack    # now includes principle library + top unresolved failures sections
    log "research-lit ingested N papers, extracted M principles, F failure patterns"
```

### Hook 2: `/idea-creator` reads AND writes wiki

**Before ideation:**
```
if research-wiki/query_pack.md exists (and < 7 days old):
    prepend query_pack to landscape context
    treat failed ideas as banlist
    treat top gaps as search seeds
    still run fresh literature search for last 3-6 months

    # NEW: run audit pass if older than 24 hours
    if AUDIT_REPORT.md is missing or > 24 hours old:
        /research-wiki audit
    include AUDIT_REPORT latent-opportunity principles in ideation seeds
    include AUDIT_REPORT OPEN contradictions as "unresolved tensions to resolve"
```

**After ideation (THIS IS CRITICAL — without it, ideas/ stays empty):**
```
for idea in all_generated_ideas (recommended + killed):
    /research-wiki upsert_idea(idea)
    for paper_id in idea.based_on:
        add_edge(idea.node_id, paper_id, "inspired_by")
    for gap_id in idea.target_gaps:
        add_edge(idea.node_id, gap_id, "addresses_gap")
rebuild query_pack
log "idea-creator wrote N ideas to wiki"
```

```
# Create experiment page

```

## Re-ideation Trigger

After significant wiki updates, suggest re-running `/idea-creator`:

- ≥5 new papers ingested since last ideation
- ≥3 new failed/partial ideas since last ideation
- New contradiction discovered in the graph
- New gap identified that no existing idea addresses

The system suggests but does not auto-trigger. User decides.

## Key Rules

- **One source of truth for relationships**: `graph/edges.jsonl`. Page `Connections` sections are auto-generated views.
- **Canonical node IDs everywhere**: `paper:<slug>`, `idea:<id>`, `exp:<id>`, `claim:<id>`, `gap:<id>`, `principle:<slug>`, `failure-pattern:<slug>`. Never use raw titles or inconsistent shorthands.
- **Failed ideas are the most valuable memory.** Never prune them from query_pack.
- **Principles are cross-project memory.** A principle extracted in project A is queryable from project B. The `tested_in_projects[]` list is how we know what has and has not been tried.
- **Failure patterns are cross-project memory.** Symmetric to principles — failures extracted in project A inform design decisions in project B. The `manifested_in_ideas[]`, `manifested_in_experiments[]` lists tell us where each failure has surfaced.
- **Principle and failure-pattern dedup is load-bearing.** Near-synonyms inflate the libraries; Layer-2/Layer-3 cosine similarity dedup keeps them finite and queryable.
- **Principles + failure patterns are symmetric.** For every principle, the `resolved_by` edges reachable from its failures form the "known defenses" against those failures. For every failure, the `affects_principles` edges form the "designs to avoid or harden" list.
- **query_pack.md is hard-budgeted** at 8000 chars. Deterministic generation, not open-ended summarization.
- **Append to log.md for every mutation.** The log is the audit trail.
- **Reviewer independence applies.** When the wiki is read by cross-model review skills, pass file paths only — do not summarize wiki content for the reviewer.
